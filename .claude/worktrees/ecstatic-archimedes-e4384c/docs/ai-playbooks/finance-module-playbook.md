# Finance Module Playbook

> Engineering reference for how the finance system works, how it should evolve, and how to add financial features safely.  
> Reference implementation: [docs/modules/accounting-finance.md](../modules/accounting-finance.md)

---

## How Logistics Finance Works

A logistics operator acts as a credit intermediary:

1. **Carrier charges the operator** — DHL/OTE bills the operator for every shipment
2. **Operator charges the customer** — The operator bills the customer at cost + markup
3. **Customer pays on account** — Customers typically pay monthly, not per-shipment
4. **Credit limit controls risk** — If a customer's unpaid balance exceeds their limit, new shipments are held

The system must track three things simultaneously:
- **What customers owe** (sum of shipment charges)
- **What customers have paid** (sum of payments received)
- **Which payments cover which shipments** (allocation)

## Current Implementation: Ledger Model

Every financial event creates an immutable row in `OrganizationLedger`. Balances are never edited — only new entries are added.

```
DEBIT  = money owed to operator  (shipment charged, COD collected on operator's behalf)
CREDIT = money received by operator or deducted from what's owed  (payment, credit note)

Balance = sum(DEBIT) - sum(CREDIT)
Positive balance = customer owes money
Zero or negative = customer is in credit
```

### Ledger Categories in Use

| Category | Type | When Created |
|---|---|---|
| `SHIPMENT_CHARGE` | DEBIT | Carrier booking confirmed |
| `PAYMENT` | CREDIT | Operator posts a received payment |
| `ALLOCATION` | CREDIT (0-amount) | Audit marker when payment is matched to shipment |
| `REVERSAL` | Opposite of original | When a ledger entry is reversed |

### Categories to Add (Not Yet Implemented)

| Category | Type | When to Create |
|---|---|---|
| `COD_COLLECTED` | CREDIT | When driver collects COD cash from receiver |
| `COD_REMITTANCE` | DEBIT | When operator remits COD cash to shipper |
| `CREDIT_NOTE` | CREDIT | When a credit note is issued (shipment damage, billing error) |
| `PENALTY` | DEBIT | Late payment fee or carrier penalty passed to customer |
| `ADJUSTMENT` | DEBIT or CREDIT | Manual correction by accounting |
| `FUEL_SURCHARGE` | DEBIT | Carrier fuel surcharge billed separately |

## COD (Cash on Delivery) Flow

COD is not currently implemented. This is how it should work:

### What COD Means in Logistics

The carrier (or operator's driver) collects cash from the receiver at delivery. That cash belongs to the shipper — the operator is just holding it temporarily. This creates a liability on the operator's side (money collected but not yet remitted to the shipper).

### COD Data Model Needed

The `Shipment` model already has `items[].value` and `declaredValue`. A `codAmount` field needs to be added:

```
Shipment:
  codAmount       Decimal?   // Amount to collect at delivery (null = not COD)
  codCurrency     String?    // Currency of COD amount
  codStatus       String?    // 'pending' | 'collected' | 'remitted' | 'failed'
  codCollectedAt  DateTime?
  codRemittedAt   DateTime?
```

### COD Ledger Flow

```
1. Shipment booked with codAmount = 50 KWD
   → No ledger entry yet (cash not yet collected)

2. Shipment delivered, driver confirms COD collected
   → OrganizationLedger: CREDIT, category: 'COD_COLLECTED', amount: 50
   → This represents: operator now holds 50 KWD belonging to the shipper

3. Operator remits COD to shipper (bank transfer, cheque)
   → OrganizationLedger: DEBIT, category: 'COD_REMITTANCE', amount: 50
   → Balance returns to neutral for this COD cycle

4. If COD not collected (receiver refused / absent)
   → codStatus = 'failed'
   → No ledger entries created
   → Shipment may need a return booking
```

### COD and Shipment Charges Together

A COD shipment has two financial flows happening simultaneously:

```
Shipment charge (what customer owes for the logistics service):
  DEBIT: SHIPMENT_CHARGE, amount: 8.500 KWD

COD collection (what operator holds on behalf of shipper):
  CREDIT: COD_COLLECTED, amount: 50.000 KWD (when delivered)
  DEBIT:  COD_REMITTANCE, amount: 50.000 KWD (when remitted)
```

These should be tracked separately — the shipment charge is revenue; the COD is a pass-through liability.

## Invoice Flow

Invoices are not currently generated by the system (labels and AWB are generated, but not customer billing invoices). This is how it should work:

### Invoice Lifecycle

```
1. Shipments are booked throughout the billing period
   → Each booking creates a SHIPMENT_CHARGE DEBIT in the ledger

2. At billing cycle end (weekly/monthly), accounting generates an invoice
   → Invoice = list of shipments + charges for the period
   → Invoice is a document, not a new ledger entry (charges are already posted)

3. Invoice is sent to customer
   → Customer pays the invoice amount

4. Payment is received and posted
   → CREDIT: PAYMENT, amount = invoice total
   → Payment is allocated to the shipments in the invoice via FIFO or manual allocation

5. If customer disputes a line item
   → CREDIT: CREDIT_NOTE, amount = disputed amount
   → Ledger entry references the original shipment
```

### Invoice Data Model Needed

The platform does not currently have an `Invoice` model. To add it:

```
Invoice:
  id                UUID
  organizationId    FK → Organization
  invoiceNumber     String (unique, sequential per org)
  periodStart       DateTime
  periodEnd         DateTime
  shipmentIds       JSON  // list of shipment IDs included
  subtotal          Decimal
  vat               Decimal
  total             Decimal
  currency          String
  status            'draft' | 'sent' | 'paid' | 'overdue' | 'disputed'
  dueDate           DateTime
  paidAt            DateTime?
  notes             Text?
  createdBy         FK → User
  createdAt         DateTime
```

Until this is built, invoices should be generated by querying `OrganizationLedger` for `SHIPMENT_CHARGE` entries in the billing period and exporting to PDF or CSV.

## Profit Tracking

The current system stores both `price` (customer charge) and `costPrice` (carrier rate) on every shipment. Profit per shipment = `price - costPrice`.

### What's Available Now

```javascript
// Per-shipment profit (from pricingSnapshot):
revenue = pricingSnapshot.totalPrice
cost    = pricingSnapshot.carrierRate + pricingSnapshot.optionalServices[].carrierAmount
markup  = pricingSnapshot.markupAmount
profit  = revenue - cost  // = markup
```

### What's Missing

**Platform-level P&L is not tracked.** To add it:

1. `costPrice` is stored per shipment but is not aggregated anywhere in the ledger
2. The `getRevenueSnapshot()` function returns revenue (DEBITs) but has no cost column
3. A `CARRIER_COST` ledger entry type would allow reconciling what DHL actually invoices vs. what was estimated at booking time

**Recommended addition:**

```
When booking is confirmed:
  → DEBIT: SHIPMENT_CHARGE, amount = customer price  (revenue)
  → DEBIT: CARRIER_COST, amount = carrier cost  (cost — tracked separately, not customer-visible)

Profit report:
  SELECT
    sum(amount) WHERE category='SHIPMENT_CHARGE' AS revenue,
    sum(amount) WHERE category='CARRIER_COST' AS cost,
    revenue - cost AS gross_profit
  FROM OrganizationLedger
  WHERE organizationId = X AND createdAt BETWEEN start AND end
```

This approach keeps the customer's ledger view (what they owe) separate from the internal cost tracking.

## Rules for Adding Finance Features

### 1. Always use `createLedgerEntry()` — never write directly to `OrganizationLedger`

`createLedgerEntry()` in `financeLedger.service.js` handles:
- Atomic balance update on `Organization`
- `balanceAfter` snapshot on the ledger row
- Transaction wrapping
- Decimal precision normalization

Never bypass this function to insert ledger rows directly.

### 2. All financial writes must be inside `prisma.$transaction()`

Any operation that modifies both a `Payment` and an `OrganizationLedger` (or both `Shipment` and `OrganizationLedger`) must be wrapped in a single Prisma transaction. If one write fails, all must roll back.

```javascript
await prisma.$transaction(async (tx) => {
    await tx.payment.create({ ... });
    await financeLedgerService.createLedgerEntry(orgId, entry, tx);  // pass tx
    await tx.organization.update({ ... });
});
```

`createLedgerEntry()` accepts an optional `externalTx` parameter for exactly this purpose.

### 3. Never delete or update ledger entries

The `OrganizationLedger` table is append-only. To correct a mistake, create a reversal entry (opposite `entryType`, `category: 'REVERSAL'`, `parentEntryId` linking to the original). The `reverseLedgerEntry()` function handles this correctly.

### 4. Use `Decimal.js` for all financial math

Never use JavaScript native `Number` arithmetic for financial calculations. Floating point errors compound. Use `decimal.js` throughout:

```javascript
const { Decimal } = require('decimal.js');
Decimal.set({ precision: 20, rounding: Decimal.ROUND_HALF_UP });

const total = new Decimal(price).plus(new Decimal(vat)).toFixed(3);
// Not: price + vat  ← floating point drift
```

### 5. API responses serialize to 3 decimal places

Use `toApiAmount()` from `financeLedger.service.js` when returning financial values. This normalizes `Decimal` instances to JavaScript numbers at 3dp precision.

### 6. The credit gate reads the cached `Organization.balance` field

The booking credit gate checks `org.balance` (the cached field), not a recomputed sum. When writing new finance features that affect the balance, always update `Organization.balance` via the atomic increment in `createLedgerEntry()`. Do not try to recompute and SET the balance from scratch — this creates a race condition.

### 7. Test allocation math with edge cases

When writing allocation tests, always cover:
- Exact payment match (payment = total charge)
- Overpayment (payment > total charge) — remainder stays as `UNAPPLIED` cash
- Partial payment (payment < total charge) — `PARTIALLY_APPLIED` status
- Allocation reversal — unapplied balance must be restored
- FIFO with multiple payments and shipments

## Known Issues in Current Implementation

See [docs/audits/audit-report.md](../audits/audit-report.md) for full details. Summary:

1. **Balance field drift risk** — `Organization.balance` is a cache maintained by atomic increments. No reconciliation job exists to detect if it drifts from ledger sums.
2. **FIFO runs outside a single transaction** — Mid-run failures leave partial allocations committed.
3. **Revenue snapshot does in-memory grouping** — Slow for large orgs; should use `$queryRaw` with SQL GROUP BY.
4. **No cancellation finance logic** — Booking a shipment and then cancelling it does not automatically reverse the `SHIPMENT_CHARGE` ledger entry.
5. **COD not implemented** — No COD amount field, no collection/remittance flow.
6. **No invoice model** — Invoices are generated externally (Excel) based on ledger data.
7. **No carrier cost tracking** — `costPrice` is stored on shipments but not tracked in the ledger, making platform P&L reporting impossible.
